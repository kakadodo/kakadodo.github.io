這個朋友必交! 關於 JavaScript 的眉眉角角 - 17.ES7 - Async 與 Await

你今天打肉了沒?
2018-12-11
JavaScript
ES7, JavaScript

自從 ES6 的 Promise 出現後，原生撰寫 Ajax 變得越來越方便，而在 ES7 版本出現了寫法更簡潔的非同步方式，就是 Async 與 Await 啦!

Async、Await 最主要的特點就是可以一次執行多個 Ajax 請求，並且請求會依照撰寫的順序一一完成。

Async 的實體其實是一個 function，不過這個 function 會回傳一個 Promise 物件，如果 function 有 return 內容，則回傳的 Promise 物件會取得 resolve 的內容，有發生錯誤的話，則是回傳 reject 的狀態及錯誤訊息 。也因為它是 function 的關係，等於是把非同步相關的程式碼都集中在一起，自然比以往更好做管理。

而 Await 在 Async function 中扮演著非常重要的角色，它會負責控管當下 Promise 請求的進度，等到當前的 Promise 回傳完成後(不論成功或失敗)，才進行下一步。這兩者可說是共生關係~~缺一不可呀!

以範例來說， getAllDatas() 就是一個 Async function，其中 async 語法要寫在 function 的最開頭讓 JavaScript 知道它與一般 function 不同，function 中則直接撰寫非同步程式，執行 Ajax 的程式開頭要加上 await 以告知這段程式需要暫停並等待完成。使用變數存放 await 的 Ajax，回傳結果會直接傳回變數身上，因此在該 function 中就可以直接取得 data，這跟以往取資料的邏輯就不太相同了~是不是更方便!!

而錯誤訊息的捕獲則是使用 try catch 語法來幫忙，將要執行的一系列 Ajax 行為寫在 try block 中，如果 Ajax 有設定發生錯誤要顯示的特殊訊息，則錯誤發生時， catch block 就會取得該 error 資訊，當然不特別設定 catch 也是會在錯誤時捕獲瀏覽器回應的資訊啦~

Async/Await 乍看可能跟 Promise.all() 很像，但其實運行方式不同，前者在執行非同步程式時是一步一步的，會等待步驟完成才繼續下一步，而 Promise.all() 則是在執行的當下就各自跑呼叫的 api，但會等到全部的 api 都回傳資訊後才一起將資料 pass 回來。

從上例就可以知道， data 的 api 呼叫會等到 gender 的 setTimeout 跑完才執行，這當中就是有 Await 在控管的關係。

不過 Async/Await 其實也可以搭配 Promise.all() 來達成同時呼叫 api。

寫來寫去好像都行的概念XDD，還是看當下的情境來決定要用甚麼方式撰寫囉…
